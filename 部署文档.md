# 部署文档

本文档说明 WY GNB PM 解析器 Rust CLI 的编译、配置和运行方式，适用于阶段 1 本地文件模式和阶段 2 FTP/SFTP 远程文件模式。

## 1. 环境要求

- 操作系统：Linux 或 macOS。
- Rust 工具链：建议使用稳定版 Rust。
- 网络：远程模式需要部署机器能访问 FTP/SFTP 服务器。
- 输入文件：当前主流程支持 CSV、CSV.gz、ZIP 中的 CSV 文件。
- CSV 格式：支持竖线分隔 position 文件，也支持 `EastCom_PM_OR...` 逗号分隔且带 header 的 LTE PRB 文件。
- 配置文件：运行目录需要包含 `mapping_dx.ini`、`rules/`、`load.toml`，远程模式还需要 `source.toml`。

检查 Rust：

```bash
rustc --version
cargo --version
```

## 2. 编译

在项目根目录执行：

```bash
cargo build --release
```

编译产物：

```text
target/release/wy-gnb-pm-parser
```

如果只做语法和依赖检查：

```bash
cargo check --release
```

## 3. 必要文件

部署目录建议包含：

```text
wy-gnb-pm-parser
mapping_dx.ini
load.toml
source.toml
rules/
```

`rules/` 目录中放 TPD 聚合规则 JSON，例如：

```text
rules/Tpd_NRCELLDU_q_5g.json
rules/TPD_NRCELLDU_PRB_Q_5G.json
rules/TPD_EUTRANCELL_0_Q.json
rules/TPD_EUTR_PRB_Q.json
rules/Tpd_NRCELLCU_q_5g.json
```

当前部署包至少应包含以下规则文件，缺少会导致对应输出表跳过：

```text
rules/TPD_EUTRANCELL_0_Q.json
rules/TPD_EUTR_PRB_Q.json
rules/TPD_NRCELLDU_PRB_Q_5G.json
rules/Tpd_NRCELLCU_q_5g.json
rules/Tpd_NRCELLDU_q_5g.json
```

## 4. 本地文件模式

本地模式使用 `--input`，可以传单个文件或目录。

单文件示例：

```bash
./wy-gnb-pm-parser \
  --input fixtures/nrcelldu_sample.csv \
  --config-dir . \
  --output-dir output/local \
  --collect-id 采集ID \
  --load-type clickhouse \
  --load-config load.toml \
  --encoding UTF-8 \
  --rules-dir rules
```

目录递归示例：

```bash
./wy-gnb-pm-parser \
  --input /data/pm/input \
  --recursive \
  --config-dir . \
  --output-dir output/local \
  --collect-id 采集ID \
  --load-type clickhouse \
  --load-config load.toml \
  --encoding UTF-8 \
  --rules-dir rules
```

## 5. 远程 FTP/SFTP 模式

远程模式使用 `--source-config source.toml`，必须同时传 `--scan-start-time`。

示例：

```bash
./wy-gnb-pm-parser \
  --source-config source.toml \
  --scan-start-time "2026-06-15 13:45:00" \
  --config-dir . \
  --output-dir output/remote \
  --collect-id 采集ID \
  --load-type clickhouse \
  --load-config load.toml \
  --encoding UTF-8 \
  --rules-dir rules
```

注意：

- `--input` 和 `--source-config` 不能同时使用。
- 远程模式缺少 `--scan-start-time` 会直接报错。
- `--scan-start-time` 格式固定为 `yyyy-MM-dd HH:mm:ss`。
- 下载后的远程文件会进入与本地模式相同的解析和输出流程。
- 下载文件会保存到 `source.toml` 中配置的 `download_dir`，同名文件会被覆盖。
- 远程扫描日志会打印每个 pattern 的扫描目录、扫描数量和匹配数量，便于定位文件缺失。

5G DU/PRB 远程示例：

```bash
./wy-gnb-pm-parser \
  --source-config source.toml \
  --scan-start-time "2026-06-17 15:15:00" \
  --config-dir . \
  --output-dir valid/remote \
  --collect-id tpd_2026051716564812850 \
  --load-type clickhouse \
  --load-config load.toml \
  --encoding UTF-8 \
  --rules-dir rules
```

LTE PRB 远程示例：

```bash
./wy-gnb-pm-parser \
  --source-config source.toml \
  --scan-start-time "2026-06-17 14:45:00" \
  --config-dir . \
  --output-dir valid/remote \
  --collect-id tpd_2026051716564812850 \
  --load-type clickhouse \
  --load-config load.toml \
  --encoding UTF-8 \
  --rules-dir rules
```

## 6. source.toml 配置

`source.toml` 示例：

```toml
[source]
type = "sftp"
download_dir = "./downloads"
remote_pattern = "/data/pm1/${SCAN_START_TIME,yyyyMMdd}/EastCom_WIRELESS_PM_ENB_EUTRANCELL_V3.*_${SCAN_START_TIME,yyyyMMddHHmm}_${SCAN_START_TIME+15m,yyyyMMddHHmm}.*.csv.zip;/data/pm2/${SCAN_START_TIME,yyyyMMdd}/EastCom_WIRELESS_PM_ENB_EUTRANCELL_V3.*_${SCAN_START_TIME,yyyyMMddHHmm}_${SCAN_START_TIME+15m,yyyyMMddHHmm}.*.csv.zip"
cache_retention_days = 7
connect_retry = 3
download_retry = 3
download_parallel = 1
retry_interval_secs = 30
connect_timeout_secs = 30
read_timeout_secs = 300

[source.connection]
host = "127.0.0.1"
port = 22
username = "user"
password = "password"
```

字段说明：

- `type`：远程类型，支持 `ftp` 或 `sftp`。
- `download_dir`：远程文件下载到本地的缓存目录。
- `remote_pattern`：远程路径正则模板。多个完整模板用英文分号 `;` 分隔。
- `cache_retention_days`：本地缓存文件保留天数，默认 7。
- `connect_retry`：连接和登录重试次数，默认 3。
- `download_retry`：单文件下载重试次数，默认 3。
- `download_parallel`：并行下载 worker 数，默认 1。大于 1 时每个 worker 独立建立 FTP/SFTP 连接。
- `retry_interval_secs`：两次重试之间等待秒数，默认 30。
- `connect_timeout_secs`：连接超时秒数，默认 30。
- `read_timeout_secs`：读写超时秒数，默认 300。

`remote_pattern` 支持以下时间模板：

- `${SCAN_START_TIME,yyyyMMdd}`
- `${SCAN_START_TIME,yyyy-MM-dd}`
- `${SCAN_START_TIME,yyyyMMddHH}`
- `${SCAN_START_TIME,yyyyMMddHHmm}`
- `${SCAN_START_TIME,yyyyMMddHHmmss}`
- `${SCAN_START_TIME+15m,yyyyMMddHHmm}`
- `${SCAN_START_TIME-15m,yyyyMMddHHmm}`
- `${SCAN_START_TIME+1h,yyyyMMddHHmm}`
- `${SCAN_START_TIME+1d,yyyyMMddHHmm}`

`remote_pattern` 渲染后作为正则表达式使用。如果要精确匹配点号，需要写成 `\\.`。

现网常用双 pattern 示例：

```toml
remote_pattern = "/data/pm/${SCAN_START_TIME,yyyyMMdd}/EastCom_WIRELESS_PM_ENB_EUTRANCELL_V3.3.0_15m_${SCAN_START_TIME,yyyyMMddHHmm}_${SCAN_START_TIME+15m,yyyyMMddHHmm}_.*.csv.(zip|gz);/data/pm/${SCAN_START_TIME,yyyyMMdd}/East(c|C)om_.*_15m_${SCAN_START_TIME,yyyyMMddHHmm}_${SCAN_START_TIME+15m,yyyyMMddHHmm}_.*.csv.(zip|gz)"
```

多目录示例：

```toml
remote_pattern = "/data/pm1/${SCAN_START_TIME,yyyyMMdd}/FILE_.*.zip;/data/pm2/${SCAN_START_TIME,yyyyMMdd}/FILE_.*.zip"
```

多目录行为：

- 每个分号片段都是一个完整远程路径正则模板。
- 某个目录不存在或扫描失败时会打印警告并继续扫描其他目录。
- 所有目录都不存在或扫描失败时，整体报错。
- 至少一个目录扫描成功但没有任何文件匹配时，整体报错并输出每个目录的扫描统计。
- 多个目录命中的远程文件会按完整远程路径排序并去重。

## 7. load.toml 配置

`load.toml` 用于生成输出包中的 `load.ctl`。

示例：

```toml
[clickhouse]
host = "${CLICKHOUSE_HOST:-127.0.0.1}"
port = 9000
database = "${CLICKHOUSE_DATABASE:-default}"
username = "${CLICKHOUSE_USER:-default}"
password = "${CLICKHOUSE_PASSWORD:-}"
table_name_case = "lower"

[postgresql]
host = "${PG_HOST:-127.0.0.1}"
port = 5432
database = "${PG_DATABASE:-postgres}"
username = "${PG_USER:-postgres}"
password = "${PG_PASSWORD:-}"
```

`load.toml` 字符串字段支持环境变量：

- `${VAR}`
- `${VAR:-default}`

## 8. 输出目录

输出包结构：

```text
{output_dir}/
  {table_lower}_{yyyyMMddHH}/
    {collect_id}_{yyyyMMddHHmm}/
      {table_lower}.csv
      {table_lower}.ini
      load.ctl
      result.csv
```

说明：

- CSV 不包含 header。
- INI 每行一个字段名。
- `load.ctl` 按 `--load-type` 生成 PostgreSQL 或 ClickHouse 加载脚本。
- Unix 平台下 `load.ctl` 自动设置为 `755`。
- `result.csv` 字段为 `table_name,data_time,row_count,success,collect_time`。

输出表名按规则表名转小写生成目录，例如：

```text
valid/remote/tpd_nrcelldu_prb_q_5g_2026061715/tpd_2026051716564812850_202606171515/
valid/remote/tpd_nrcelldu_q_5g_2026061715/tpd_2026051716564812850_202606171515/
valid/remote/tpd_nrcellcu_q_5g_2026061715/tpd_2026051716564812850_202606171515/
valid/remote/tpd_eutr_prb_q_2026061714/tpd_2026051716564812850_202606171445/
```

## 9. 性能和解析策略

当前版本包含以下流式优化：

- TPD 规则默认走流式聚合，不再把所有源表 rows 全量保存在内存中。
- 同一 source table、相同 group_by、相同 where 的规则共享聚合状态，例如 `Tpd_NRCELLDU_q_5g` 和 `TPD_NRCELLDU_PRB_Q_5G`。
- `OP_NRCELLDU` 启用 values fast path，position 文件按规则依赖字段直接解析为字段数组，避免大宽表逐字段构造完整 `Row`。
- position 文件会跳过明显 header 行，避免将 `SCAN_START_TIME`、`PHY_...` 等列名当作数据。
- 5G DU 场景会从文件名兜底 `scan_start_time` / `scan_stop_time`，文件名时间支持 `yyyyMMddHHmm` 和 `yyyyMMddHHmmss`。
- LTE PRB 的 `EastCom_PM_OR...` 文件按逗号分隔、header CSV 解析。
- DN 派生字段按需拆解；规则需要 `enbfunction`、`managedelement`、`parent_dn` 等字段时会从 `DN` 中派生。

已验证场景参考：

```text
2026-06-17 15:15:00 5G CU/DU/DU PRB：约 40s，总输出 3 张 5G 表。
2026-06-17 14:45:00 LTE PRB：约 60s，输出 TPD_EUTR_PRB_Q。
2026-06-15 13:45:00 EUTRANCELL：正常输出 TPD_EUTRANCELL_0_Q。
```

日志中出现 `fast=values` 表示命中了宽表 fast path，例如：

```text
-> table=OP_NRCELLDU rows=84079 cols=900 counter=NRCELLDU filenum=-1 fast=values
```

## 10. 常见问题

### 远程模式提示缺少 scan-start-time

远程模式必须传：

```bash
--scan-start-time "2026-06-15 13:45:00"
```

### 远程扫描到 0 个匹配文件

检查：

- `--scan-start-time` 是否正确。
- `remote_pattern` 日期格式是否正确。
- 多目录配置中是否使用英文分号 `;` 分隔。
- 正则中的点号是否需要转义。
- 推导出的扫描目录是否过大或过小。
- 远程目录下实际文件名是否符合规则。

### 连接失败或登录失败

检查：

- `host`、`port` 是否正确。
- 部署机器是否能访问远程服务器。
- `username`、`password` 是否正确。
- FTP/SFTP 服务端是否允许当前认证方式。

### 下载后解析失败

检查：

- 文件是否为当前支持的 CSV、CSV.gz 或 ZIP 中 CSV。
- `mapping_dx.ini` 是否包含对应 counter/table 映射。
- `rules/` 中是否包含对应 TPD 规则。
- 输入文件编码是否与 `--encoding` 一致。

### LTE PRB 提示 missing field enbfunction

检查：

- 文件名是否为 `EastCom_PM_OR...`。该类文件是逗号分隔 header CSV。
- 部署版本是否包含逗号 header 解析修复。
- `rules/TPD_EUTR_PRB_Q.json` 是否存在。
- `mapping_dx.ini` 是否包含 `OP_EUTRANCELLFDDPRB` / `OP_EUTRANCELLTDDPRB` 的 `DN`、`RDN`、`PARENT_DN`、`SOURCEFILENAME` 映射。

### 5G DU PRB 解析慢或卡在 parse

检查日志中 `OP_NRCELLDU` 是否显示 `fast=values`。

如果没有显示：

- 确认 `rules/TPD_NRCELLDU_PRB_Q_5G.json` 存在。
- 确认 `rules/Tpd_NRCELLDU_q_5g.json` 存在。
- 确认规则 group_by 为 `dn, scan_start_time, scan_stop_time`。
- 确认 source table 为 `OP_NRCELLDU`。

### 输出提示 invalid scan_start_time

检查：

- position 文件是否误把 header 当数据。
- 文件名是否包含扫描起止时间，例如 `202606171515_202606171530`。
- `scan_start_time` / `scan_stop_time` 是否能从源列或文件名兜底得到。

## 11. 验证命令

部署前建议执行：

```bash
cargo test --workspace
cargo clippy --workspace --all-targets --all-features
cargo check --release
cargo build --release
```

涉及 Java 的场景只做静态语法验证，不要求在部署机上编译或运行 Java 参考实现。

如果只部署二进制，不需要在生产机器运行测试；生产机器只需保证二进制和配置文件齐全。

## 12. 安全注意事项

- 不要把真实 FTP/SFTP 密码提交到 Git 仓库。
- `source.toml` 中的密码当前不支持环境变量插值，生产环境应通过部署流程管理该文件权限。
- 建议限制 `source.toml` 权限，例如仅运行用户可读。
- 输出目录和下载缓存目录不要与系统目录混用。
