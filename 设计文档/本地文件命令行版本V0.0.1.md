# 数据采集逻辑说明

本文档说明当前 Rust PM 解析工具的数据采集、转换和输出流程。

## 整体流程

CLI 按以下四个阶段处理输入文件：

1. 从 `--config-dir` 加载 `mapping_dx.ini`。
2. 从 `--input` 收集并解析 PM 输入文件，写入内存中的 OP 表。
3. 执行 `--rule-file` 和/或 `--rules-dir` 指定的 TPD 聚合规则。
4. 将非 OP 结果表写出为 CSV 文件到 `--output-dir`。

主流程在 `src/main.rs` 中实现。

## 运行参数

主要 CLI 参数如下：

- `--input`：输入文件或输入目录。
- `--config-dir`：包含 `mapping_dx.ini` 的目录，默认是当前目录。
- `--output-dir`：生成 CSV 的输出目录。
- `--collect-id`：采集批次标识，用于输出包第二层目录前缀。
- `--load-type`：加载脚本类型，支持 `postgresql` 和 `clickhouse`。
- `--load-config`：加载脚本配置 TOML 路径，默认 `load.toml`。
- `--output-delimiter`：输出文件分隔符，默认是 `|`；当前要求是单字节 ASCII 字符。
- `--encoding`：输入文本编码，默认 UTF-8；当前也支持 GBK/GB2312 解码。
- `--recursive`：当 `--input` 是目录时，递归收集子目录文件。
- `--rule-file`：一个或多个 JSON TPD 规则文件。
- `--rules-dir`：包含 JSON TPD 规则的目录；目录内 `.json` 文件会按文件名排序后加载。

## 映射配置

`mapping_dx.ini` 驱动源表识别和字段映射。

当前解析两个 section：

- `[TableMapping]`：将文件名中的 counter 名称映射到 OP 表名。
- `[ColNameMapping]`：将源字段映射到目标 OP 表字段。

全局 `filenum` 控制 CSV 解析模式：

- `filenum = -1`：无表头模式。按配置中的字段顺序做位置映射。
- 其他值：有表头模式。按规范化后的表头名称做字段映射。

映射层会将表名规范化为大写。

## 输入文件收集

如果 `--input` 是文件，则只处理该文件。

如果 `--input` 是目录：

- 未指定 `--recursive` 时，只处理该目录下的直接子文件。
- 指定 `--recursive` 时，处理目录树下的所有文件。

文件收集阶段不按扩展名过滤，具体格式判断在解析阶段完成。

## 文件格式处理

解析逻辑在 `src/parser.rs` 中实现。

当前支持的输入处理方式：

- `.gz`：先解压到临时文件，再按解压后的文件类型继续解析。
- `.zip`：将每个非目录条目解压到临时目录，再逐个解析。
- `.csv`：按分隔符文本解析。
- 其他文件：如果前 512 字节包含 `|`，也按分隔符文本解析。

XML 文件当前会报错：`XML parsing is not implemented yet`。其他未知文件类型会被忽略。

临时解压文件位于进程级临时目录中，进程退出时清理。

## CSV 分隔符选择

分隔符选择保持当前 Java 兼容行为：

- 文件名以 `EastCom_PM_OR` 开头时，使用逗号 `,`，并强制 `filenum = 0`。
- 其他文件使用管道符 `|`，并使用 `mapping_dx.ini` 中的 `filenum`。

## Counter 和 OP 表识别

每个 CSV 文件按以下步骤识别源表：

1. 将文件名转为大写。
2. 按 key 长度倒序扫描 `[TableMapping]` 中的 key。
3. 选择第一个被文件名包含的 key 作为 counter。
4. 通过 counter 映射到基础 OP 表。

基础表识别后会应用 SA/NSA 覆盖逻辑：

- 如果文件名包含 `-NSA-` 或 `_NSA_`，检查表映射中是否存在 `{BASE_TABLE}_NSA`。
- 如果文件名包含 `-SA-` 或 `_SA_`，检查表映射中是否存在 `{BASE_TABLE}_SA`。
- 如果没有对应覆盖表，则使用基础 OP 表。

## 行数据构建

每一行解析结果都存储为有序 key-value map。

无表头模式（`filenum = -1`）：

- 使用 `[ColNameMapping]` 中配置的目标字段顺序。
- 输入行按分隔符切分后，按下标赋值。
- 缺失值填为空字符串。

有表头模式：

- CSV 表头会经过 `column_name_format` 和大写转换进行规范化。
- 源数据按规范化后的表头名称存储。
- 每个配置的源字段会通过候选名称查找实际值。
- 查到的值写入配置的目标字段名。

字段值会做以下规范化：

- `/0` 转为 `0`。
- `NIL`、`NULL`、`N/A`、`-`、`""` 转为空字符串。
- 其他值中的双引号会被移除。
- 其他值会 trim 后保留。

## 行数据增强

字段映射完成后，每一行会补充派生字段。

`is_nsa` 处理：

- 有表头模式下，如果源数据包含 `IS_NSA`，直接使用该值。
- 否则文件名包含 `-NSA-` 或 `_NSA_` 时，设置 `is_nsa = 1`。
- 文件名包含 `-SA-` 或 `_SA_` 时，设置 `is_nsa = 0`。

DN 拆分：

- 解析器会查找 `Dn`、`dn` 或源字段 `DN`。
- 如果存在 DN，则按逗号拆分。
- `key=value` 形式的片段会同时插入原始 key 和小写 key。
- `parent_dn` 会被设置为最后一个逗号之前的内容。

## 内存表模型

所有解析出的 OP 行都存放在内存中，结构为：

```text
HashMap<table_name, Vec<Row>>
```

`Row` 是字段名到字符串值的有序 map。

OP 表是聚合的中间源表，不会写出到最终 CSV。

## TPD 规则加载

TPD 规则是从 Excel 规则定义导出的 JSON 文件。

每个规则包含：

- `table_name`：输出 TPD 表名。
- `groups`：分组和源表配置。
- `temp_fields`：输出字段计算前的中间字段。
- `output_fields`：最终输出字段和表达式。

使用 `--rules-dir` 时，目录内所有 `.json` 文件会按排序后追加到显式 `--rule-file` 列表之后执行。

## TPD 聚合

聚合逻辑在 `src/tpd.rs` 中实现。

每个规则按以下步骤执行：

1. 选择第一个 enabled 的 group。
2. 在内存 OP 表中查找配置的源表。
3. 如果源表不存在，打印日志并跳过该规则。
4. 按配置的 `group_by` 字段对源行分组。
5. 对每个分组，将 group-by 字段值写入 context。
6. 依次计算 `temp_fields`，并写入 context。
7. 依次计算 `output_fields`，并写入输出行。
8. 将生成的行存入大写输出表名下。

当前实现会打印聚合耗时，包含 grouping、temp 计算、output 计算、overhead 和 total。

## 表达式支持

Rust 实现不调用 SQL 引擎，而是在代码中实现当前 JSON 规则需要的表达式子集。

当前支持：

- `max(field)` 和 `max(expression)`。
- `count(distinct field)`。
- `CRC64(expression)`。
- 使用 `||` 的字符串拼接。
- 字符串字面量，例如 `'8105:'`；这些值来自规则 JSON，不在 Rust 代码中写死。
- 环境变量占位符，例如 `'${ENV_NAME}'`。
- `CURRENT_TIMESTAMP`。
- 简单 `case when count_field > number then expr else expr end`。
- `substring(expression,start,length)`。
- `locate(needle,haystack)`，位置按 SQL 风格从 1 开始。
- `length(expression)`。
- 数值参数中的 `+` 和 `-` 运算。

为了性能，简单 `max(field)` 会走直接字段查找路径；复杂表达式，例如 `max(substring(...))`，才走表达式求值器。

## CRC64

`CRC64(...)` 使用 `src/crc64.rs` 中的 Java 兼容 CRC 实现。

返回值格式与参考 Java 实现一致，是 signed integer 字符串。典型规则包括 `CRC64(max(object_rdn))` 和 `CRC64('8104:'||parent_dn)`。

## 输出写入

输出逻辑在 `src/writer.rs` 中实现。当前输出为采集包结构：

```text
output/
  {table_lower}_{yyyyMMddHH}/
    {collect_id}_{yyyyMMddHHmm}/
      load.ctl
      result.csv
      {table_lower}.csv
      {table_lower}.ini
```

写出规则如下：

- 表名以 `OP_` 开头的表会跳过。
- 非 OP 表按 `scan_start_time` 拆分采集包。
- CSV 文件名为 `{table_lower}.csv`，不写表头。
- INI 文件名为 `{table_lower}.ini`，每行一个字段名。
- `load.ctl` 根据 `--load-type` 生成 PostgreSQL 或 ClickHouse 加载脚本。
- `load.ctl` 生成后会在 Unix 平台设置为可执行权限 `755`。
- `result.csv` 记录 `table_name,data_time,row_count,success,collect_time`，其中 `success` 为 `1` 表示成功。
- ClickHouse 加载脚本中的表名大小写由 `load.toml` 的 `clickhouse.table_name_case` 控制，支持 `upper` 和 `lower`，默认是 `lower`；非法值会报错。
- `load.toml` 字符串字段支持环境变量引用，例如 `${CLICKHOUSE_HOST}` 和 `${CLICKHOUSE_HOST:-127.0.0.1}`。
- 输出分隔符由 `--output-delimiter` 控制，默认是 `|`。
- 如果目标路径已有同名文件，会覆盖。
- 如果 `mapping_dx.ini` 中配置了该表头，则使用配置表头。
- 如果没有配置表头，则从输出行 key 顺序推断表头。
- CSV 输出使用 Rust `csv` writer，文本为 UTF-8。

## 当前限制

- XML 解析尚未实现。
- 所有行会在内存中保留，直到最终输出完成。
- 表达式支持范围有意限制为当前 JSON 规则使用的子集。
- `where_expr`、`order_by`、`join_keys` 当前会解析，但没有完整执行 SQL 语义。
- 规则源表不存在时会跳过并打印日志，而不是作为致命错误。

## 示例命令

```bash
cargo run --release -- \
  --input fixtures/EastCom_WIRELESS_PM_ENB_EUTRANCELL_V3.3.0_15m_202606151345_202606151400_P002.csv \
  --config-dir . \
  --output-dir output \
  --collect-id 采集ID \
  --load-type clickhouse \
  --load-config load.toml \
  --output-delimiter '|' \
  --encoding UTF-8 \
  --rule-file rules/TPD_EUTRANCELL_0_Q.json
```

该命令会将 EUTRANCELL PM CSV 解析为 `OP_EUTRANCELL`，执行 `TPD_EUTRANCELL_0_Q.json`，并按采集包结构写出 CSV、INI、`load.ctl` 和 `result.csv`。
