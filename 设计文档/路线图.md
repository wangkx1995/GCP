# 项目路线图

本文档整理当前项目的实施顺序，分为本地 CLI 基线、FTP/SFTP 远程输入、平台化扩展三阶段。

当前状态：阶段 1 已冻结，下一步进入阶段 2：FTP/SFTP 远程输入。

## 一、总体原则

- 先稳住当前本地 CLI 基线，再扩展远程输入，最后再做平台化。
- 每一阶段都要有明确边界、交付物和验收标准。
- 不引入 SQL 引擎，表达式继续由 Rust 自己实现。
- 远程输入优先复用现有本地解析链路，不重复造轮子。

## 二、阶段 0：基线冻结

### 目标

把当前本地 CLI 版本作为唯一稳定基线。

### 范围

- 本地文件输入
- `mapping_dx.ini` 配置
- OP 表解析
- TPD 规则聚合
- CSV 输出

### 交付物

- 当前实现说明文档
- 样本输入输出对照结果
- 基线性能数据

### 验收标准

- 样本可稳定跑通
- 输出结果与 Java 参考一致
- 当前性能满足单机本地使用

## 三、阶段 1：本地 CLI 完整化

状态：已冻结。冻结记录见 `阶段1冻结记录.md`。

### 目标

把本地 CLI 做成可独立交付的稳定版本。

### 重点工作

- 补齐输入格式、编码、分隔符说明
- 持续优化表达式执行性能
- 保持输出格式可配置
- 完善错误日志和边界行为

### 交付物

- `本地文件命令行版本V0.0.1.md`
- `阶段1冻结记录.md`
- 性能和样本验证记录

### 验收标准

- 本地单机版可独立使用
- 规则结果稳定
- 性能可接受，且可观测

## 四、阶段 2：FTP/SFTP 远程输入

### 目标

在不改动现有解析核心的前提下，增加远程文件输入能力。

### 功能顺序

1. 目录扫描
2. 正则匹配
3. 下载到本地缓存
4. 复用现有解析链路
5. 下载和扫描错误处理

### 交付物

- 远程输入模块设计
- 下载与缓存策略
- 失败处理和重试策略

### 验收标准

- FTP/SFTP 文件最终能落到本地临时目录
- 落地后的文件进入同一套本地解析流程
- 失败时有清晰日志和可排障信息

## 五、阶段 3：平台化扩展

### 目标

在稳定引擎基础上，再做统一调度、统一配置和 Web 管理平台。

### 方向

- DB / Kafka / RESTful 适配器
- Web 配置管理平台
- 任务调度与监控
- 统一错误和日志视图

### 交付物

- 平台架构设计
- 任务和配置模型
- 管理端页面和接口

### 验收标准

- 各类数据源共用统一调度模型
- 配置可视化管理
- 任务状态可监控、可追踪

## 六、风险与约束

- 不引入 SQL 引擎。
- XML 解析暂不作为当前阶段重点。
- 远程输入先复用现有解析链路。
- 平台化不要和本地 CLI 混做。

## 七、推荐执行顺序

1. 冻结阶段 0。
2. 阶段 1 已冻结。
3. 实施阶段 2。
4. 最后推进阶段 3。
